# 陈平(测试工程师) Notion 看板使用规范

> **适用范围**: 陈平(测试工程师)  
> **维护责任**: 陈平 与 陆贾(知识库管理员) 共同维护 Bug 看板  
> **关联角色**: 张良(产品经理)、韩信(全栈研发)、萧何(技术负责人)

## 一、Bug 看板结构

### 1.1 数据库名称
`🐛 Bug 追踪看板` (Database)

### 1.2 视图配置

| 视图名称 | 类型 | 用途 |
|---------|------|------|
| 状态看板 | Board | 按状态分组，日常跟踪 |

## 二、Bug 单字段填写规范

### 2.1 基础字段

| 字段名 | 类型 | 必填 | 填写规范 |
|--------|------|------|----------|
| **Bug 编号** | Title | ✅ | 自动生成或按 `BUG-YYYYMMDD-序号` 格式 |
| **状态** | Select | ✅ | 新增 / 重复打开 / 修复中 / 已修复 / 非Bug / 需求设计问题 |
| **严重级别** | Select | ✅ | P0-致命 / P1-严重 / P2-一般 / P3-轻微 / P4-建议 |
| **优先级** | Select | ✅ | 紧急 / 高 / 中 / 低 |
| **指派给** | Person | ✅ | 当前处理人 |
| **报告人** | Person | ✅ | 发现人 |
| **所属模块** | Select | ✅ | 与产品模块对应 |
| **迭代版本** | Select | ✅ | 关联当前迭代 |

### 2.2 缺陷描述字段

| 字段名 | 类型 | 必填 | 填写规范 |
|--------|------|------|----------|
| **缺陷标题** | Text | ✅ | 简明描述问题，格式：`[模块]问题概述` |
| **复现步骤** | Text | ✅ | 1. 步骤一 2. 步骤二 3. 预期结果 4. 实际结果 |
| **环境信息** | Text | ✅ | 浏览器/设备/系统版本/测试环境 |
| **截图/录屏** | Files | 建议 | 必须包含错误截图，复杂流程配录屏 |
| **日志附件** | Files | 可选 | 错误日志、控制台输出 |
| **影响范围** | Text | 可选 | 影响用户/功能/数据的具体范围 |

### 2.3 流程字段

| 字段名 | 类型 | 用途 |
|--------|------|------|
| **创建时间** | Date | 自动记录 |
| **修复时间** | Date | 开发标记已修复时填写 |
| **验证时间** | Date | 测试验证完成时填写 |
| **关闭时间** | Date | Bug关闭时填写 |
| **回归次数** | Number | 重复打开次数统计 |
| **关联需求** | Relation | 关联需求文档 |
| **关联PR** | URL | 关联代码PR链接 |
| **验收结果** | Select | 待验收 / 通过 / 不通过 |
| **不通过原因** | Text | 验收失败时填写原因 |

## 三、Bug 状态流转规则

### 3.1 状态定义

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  新增   │ → │ 修复中  │ → │  已修复  │ → │  已关闭  │
└────┬────┘    └─────────┘    └────┬────┘    └─────────┘
     │                              │
     │         ┌─────────┐          │
     └────────→│重复打开 │←─────────┘
               └─────────┘
     
     ↓
┌─────────┐    ┌─────────┐
│  非Bug  │    │需求设计 │
│         │    │  问题   │
└─────────┘    └─────────┘
```

### 3.2 状态流转说明

| 流转方向 | 触发条件 | 操作人 | 备注 |
|---------|---------|--------|------|
| 新增 → 修复中 | 开发认领Bug | 韩信 | - |
| 修复中 → 已修复 | 开发提交修复 | 韩信 | 必须填写修复说明和关联PR |
| 已修复 → 已关闭 | 测试验证通过 | 陈平 | 验证无误后关闭 |
| 已修复 → 重复打开 | 测试验证不通过 | 陈平 | 不通过原因必须填写 |
| 新增 → 非Bug | 确认为误报 | 陈平/陆贾 | 需注明非Bug原因 |
| 新增 → 需求设计问题 | 属设计缺陷 | 陈平 | 转给张良处理 |
| 已修复 → 重复打开(产品) | 产品验收失败 | 张良 | 17:30验收流程 |

### 3.3 严重级别定义

| 级别 | 定义 | 响应时效 | 验收标准 |
|------|------|---------|---------|
| P0-致命 | 系统崩溃/数据丢失/核心功能不可用 | 2小时内修复 | 阻塞发版 |
| P1-严重 | 主要功能异常/明显错误 | 24小时内修复 | 必须修复 |
| P2-一般 | 次要功能异常/有替代方案 | 迭代内修复 | 视情况修复 |
| P3-轻微 | UI偏差/文案错误 | 下个迭代 | 可选修复 |
| P4-建议 | 优化建议/体验改进 | 排期评估 | 不阻塞发版 |

## 四、产品经理验收协作流程

### 4.1 验收触发条件

- **时间**: 每日 17:30 统一执行
- **范围**: 当日标记为"已修复"的Bug
- **执行人**: 张良(产品经理)

### 4.2 验收流程

```
┌─────────────────────────────────────────────────────────────┐
│  17:30 验收检查清单                                          │
├─────────────────────────────────────────────────────────────┤
│  1. 筛选今日"已修复"状态Bug列表                              │
│  2. 张良逐一验证功能是否符合预期                             │
│  3. 验证通过 → 保持"已修复" → 陈平次日回归测试               │
│  4. 验证不通过 → 状态改为"重复打开"                          │
│              → 填写"不通过原因"                             │
│              → 指派给相应开发                                │
│              → 通知曹参更新进度风险                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 验收失败处理

| 验收失败场景 | 操作 | 通知对象 |
|-------------|------|---------|
| 功能未修复 | 状态→重复打开，指派开发 | 韩信、曹参 |
| 修复不完全 | 状态→重复打开，补充说明 | 韩信、曹参 |
| 引入新问题 | 状态→重复打开，新增关联Bug | 韩信、曹参、萧何 |
| 与设计不符 | 状态→需求设计问题 | 张良、叔孙通 |

## 五、每日定时任务 (14:00 Bug单更新)

### 5.1 定时任务触发

- **时间**: 每日 14:00
- **执行人**: 陈平
- **任务来源**: 系统定时提醒 / OpenViking 触发

### 5.2 操作清单

| 序号 | 操作项 | 具体操作 | 输出 |
|------|--------|---------|------|
| 1 | 检查"修复中"Bug | 查看开发进度，确认是否按预期推进 | 进度备注 |
| 2 | 更新Bug状态 | 将已修复但未验证的Bug标记待验证 | 状态更新 |
| 3 | 清理超时Bug | 对超过48小时无响应的Bug升级 | 提醒通知 |
| 4 | 整理验证列表 | 准备当日/次日验证清单 | 验证计划 |
| 5 | 同步陆贾 | 与知识库管理员同步看板状态 | 协同确认 |

### 5.3 Notion 看板操作步骤

```
1. 打开 Notion Bug 看板
2. 筛选"修复中"状态的Bug
3. 按"指派给"分组，确认每个Bug进度
4. 对超时Bug添加评论：@韩信 请更新进度
5. 筛选"已修复"状态的Bug
6. 按优先级排序，制定验证顺序
7. 在今日待验证的Bug中添加"今日验证"标签
8. 与陆贾同步：评论标记"@陆贾 今日Bug状态已更新"
```

## 六、回归测试流程

### 6.1 触发条件

- Bug 标记为"已修复"且产品验收通过
- 版本发布前的全量回归

### 6.2 回归测试看板操作

| 步骤 | 操作 | Notion字段更新 |
|------|------|---------------|
| 1 | 按模块筛选已修复Bug | 视图：回归测试清单 |
| 2 | 执行验证测试 | 添加验证备注 |
| 3 | 验证通过 | 状态→已关闭，填写验证时间 |
| 4 | 验证不通过 | 状态→重复打开，回归次数+1 |
| 5 | 统计回归率 | 更新迭代质量报告 |

## 八、常用 Notion 筛选器配置

### 8.1 保存的筛选器

```
📌 今日待验证
   状态 = 已修复
   AND 指派给 = 陈平
   AND 修复时间 = 今天

📌 我负责的P0/P1
   指派给 = 陈平
   AND 严重级别 IN (P0-致命, P1-严重)
   AND 状态 != 已关闭

📌 重复打开Bug
   回归次数 > 0
   AND 状态 != 已关闭
   排序：回归次数 DESC

📌 本周待回归
   状态 = 已修复
   AND 修复时间 >= 本周一
```
