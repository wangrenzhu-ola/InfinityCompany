## 一、外部需求看板读取权限与字段

### 1.1 看板访问权限

| 权限类型 | 权限范围 | 说明 |
|---------|---------|------|
| **读取权限** | Notion 外部需求看板 | 09:15 定时读取待审事项 |
| **读取权限** | 迭代看板 | 同步任务状态，了解分发后进展 |
| **更新权限** | Kanban Task 状态 | 更新需求处理状态 |
| **创建权限** | 意图分发记录 | 在指定数据库创建分发日志 |

### 1.2 外部需求看板字段定义

**看板名称**：`外部需求看板 (External Requests)`

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|-----|------|
| `需求ID` | Title | ✅ | 唯一标识，格式：`EXT-YYYYMMDD-NNN` |
| `来源` | Select | ✅ | 外部用户/同事/合作方/其他 |
| `原始描述` | Text | ✅ | 外部用户提交的原始需求描述 |
| `登记时间` | Date | ✅ | 郦食其登记到看板的时间 |
| `登记人` | Person | ✅ | 郦食其（外部助理） |
| `状态` | Status | ✅ | 待识别 → 待审核 → 已批准 / 已拒绝 / 需补充 |
| `意图分类` | Select | - | 工作类 / 生活类 / 紧急类 / 待识别 |
| `建议分发` | Multi-select | - | 张良(产品) / 曹参(PMO) / 韩信(技术) / 健康管理 |
| `审核备注` | Text | - | Owner审核时的决策备注 |
| `关联Task` | Relation | - | 批准后创建的关联任务 |
| `优先级` | Select | - | P0(紧急) / P1(重要) / P2(一般) / P3(低) |

### 1.3 状态流转规则

```
郦食其登记
    │
    ▼
┌─────────┐    夏侯婴09:15读取    ┌─────────┐    Owner审核    ┌─────────┐
│  待识别  │ ───────────────────→ │  待审核  │ ────────────→ │  已批准  │
└─────────┘                      └─────────┘               └─────────┘
                                                                      │
    ┌─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────┐    ┌─────────┐    ┌─────────┐
│  已拒绝  │    │  需补充  │    │ 已分发  │
└─────────┘    └─────────┘    └─────────┘
```

## 二、意图识别后的分发记录方式 (dispatch.log)

### 2.1 分发记录数据库

**数据库名称**：`意图分发日志 (Intent Dispatch Log)`

| 字段名 | 类型 | 必填 | 说明 |
|-------|------|-----|------|
| `记录ID` | Title | ✅ | 唯一标识，格式：`DISP-YYYYMMDD-NNN` |
| `时间戳` | DateTime | ✅ | 分发决策时间 |
| `源需求ID` | Relation | ✅ | 关联外部需求看板的需求ID |
| `意图分类` | Select | ✅ | 工作类 / 生活类 / 紧急类 |
| `目标角色` | Multi-select | ✅ | 张良 / 曹参 / 韩信 / 陆贾 / 健康管理 |
| `分发理由` | Text | ✅ | 为何如此分发的简要说明 |
| `Owner决策` | Select | - | 待确认 / 批准 / 拒绝 / 需调整 |
| `执行状态` | Status | ✅ | 待执行 / 已通知 / 已接受 / 已完成 |
| `创建人` | Person | ✅ | 夏侯婴 |

### 2.2 分发记录格式规范

**每条记录必须包含**：

```markdown
## 分发记录 [DISP-20260327-001]

- **时间戳**: 2026-03-27 09:15:30
- **源需求**: EXT-20260327-003
- **意图分类**: 工作类 - 产品需求
- **建议分发**: 张良(产品经理)
- **理由**: 涉及新功能定义，需产品负责人进行需求澄清和PRD编写
- **Owner决策**: 待确认
- **执行状态**: 待执行
```

### 2.3 意图分类标准

| 分类 | 判定标准 | 建议分发目标 |
|-----|---------|-------------|
| **工作类-产品** | 涉及功能定义、用户体验、需求澄清 | 张良(产品经理) |
| **工作类-技术** | 涉及系统架构、技术实现、代码开发 | 韩信(全栈研发) + 萧何(架构师) |
| **工作类-迭代** | 涉及排期调整、资源协调、进度同步 | 曹参(PMO) |
| **工作类-外部** | 涉及对外沟通、客户对接 | 郦食其(外部助理) |
| **生活类-健康** | 涉及饮食、运动、作息、体检 | 健康管理链路 |
| **紧急类** | 阻塞当前迭代或影响系统稳定性 | 直接向Owner汇报，同步通知相关角色 |

### 3.2 Owner 决策同步回 Notion

**决策录入方式**：

| 决策结果 | Notion 操作 | 后续动作 |
|---------|------------|---------|
| **批准** | 更新需求状态为「已批准」，填写「审核备注」 | 夏侯婴按建议分发，创建关联Task |
| **拒绝** | 更新需求状态为「已拒绝」，填写拒绝理由 | 通知郦食其向外部说明 |
| **需补充** | 更新需求状态为「需补充」，注明补充要求 | 退回郦食其补充信息 |
| **调整** | 更新「建议分发」和「优先级」字段 | 按调整后方案执行 |

**审核备注模板**：
```
[2026-03-27 09:20] Owner决策：
- 批准 EXT-003，优先级降为P2，下迭代处理
- 批准 EXT-004，按P0立即处理
- 拒绝 EXT-005，建议通过其他渠道处理
```

## 五、09:15 定时任务的具体看板操作步骤

### 5.1 操作前准备

1. 确保 Notion 访问权限正常
2. 确认昨日 Owner 审核决策已同步
3. 检查昨日分发任务的执行状态

### 5.2 详细操作步骤

```
步骤 1: 读取外部需求看板
├── 打开 Notion → 外部需求看板
├── 筛选条件：状态 = "待识别"
├── 按「登记时间」升序排列（先登先处理）
└── 记录今日待识别需求数量：___ 项

步骤 2: 逐条进行意图识别
├── 读取「原始描述」
├── 判断意图分类：工作类 / 生活类 / 紧急类
├── 判定标准：
│   ├── 涉及功能/产品 → 工作类-产品
│   ├── 涉及技术/bug → 工作类-技术
│   ├── 涉及排期/资源 → 工作类-迭代
│   ├── 涉及饮食/运动/作息 → 生活类-健康
│   └── 阻塞当前迭代 → 紧急类
└── 填写「意图分类」字段

步骤 3: 生成分发建议
├── 工作类-产品 → 建议分发：张良
├── 工作类-技术 → 建议分发：韩信 + 萧何
├── 工作类-迭代 → 建议分发：曹参
├── 生活类-健康 → 处理方式：健康管理链路
└── 紧急类 → 处理方式：直报Owner + 同步相关角色

步骤 4: 整理待审核清单
├── 创建页面：「待审核事项清单 - YYYY-MM-DD」
├── 分类汇总：工作类 / 生活类 / 需决策
├── 标注建议优先级和理由
└── 设置截止时间：09:30（晨会前）

步骤 5: 提交 Owner 审核
├── 通过即时通讯发送清单链接
├── 提醒审核截止时间
└── 等待 Owner 决策

步骤 6: 同步决策结果（Owner批复后）
├── 读取 Owner 审核意见
├── 更新每条需求的「状态」字段
├── 填写「审核备注」
├── 如需调整，更新「建议分发」和「优先级」
└── 进入「分发执行」流程

步骤 7: 执行分发
├── 对「已批准」的需求：
│   ├── 在「意图分发日志」创建记录
│   ├── 通知对应角色（创建Task或发送消息）
│   ├── 在外部需求看板更新「执行状态」
│   └── 创建「关联Task」关系
├── 对「已拒绝」的需求：
│   └── 通知郦食其向外部说明
└── 对「需补充」的需求：
    └── 退回郦食其，注明补充要求

步骤 8: 完成确认
├── 检查所有「待识别」项是否已处理
├── 确认分发记录已完整创建
├── 更新今日处理统计
└── 准备 09:25 晨会提醒
```

### 5.3 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| Notion 访问失败 | 立即向 Owner 报告，尝试从缓存或邮件获取需求 |
| 需求描述不清 | 标记「需补充」，退回郦食其澄清 |
| 意图难以判定 | 列入「需决策」事项，请 Owner 指示 |
| Owner 未及时审核 | 09:25 发送提醒，如 09:30 仍未批复，带入晨会讨论 |
| 分发对象不接受 | 记录原因，退回 Owner 重新决策 |

## 六、Owner决策同步回Notion的更新流程

### 6.1 同步时机

- **即时同步**：Owner 做出决策后立即更新
- **批量同步**：如 Owner 批量批复，09:30 前统一更新
- **延迟同步**：Owner 延迟决策的，批复后 5 分钟内更新

### 6.2 状态更新映射

| Owner 口头决策 | Notion 状态更新 | 附加操作 |
|---------------|----------------|---------|
| "这个可以做" | 状态 → 「已批准」 | 创建分发记录 |
| "这个不做" | 状态 → 「已拒绝」 | 填写拒绝理由 |
| "先放着" | 状态 → 「待审核」+ 优先级调整 | 添加备注说明 |
| "让 XX 先评估" | 状态 → 「待审核」+ 建议分发更新 | 通知评估人 |
| "下周再说" | 状态 → 「待审核」+ 优先级降为P3 | 设置提醒 |

### 6.3 审核备注填写规范

```markdown
[时间] 决策者：决策内容

示例：
[2026-03-27 09:20] 刘邦：
- 批准，按建议分发给张良
- 优先级调整为P2，不进入本迭代
- 备注：需先进行用户调研
```

### 6.4 关联 Task 创建规则

当 Owner 批准后，需在对应角色看板创建 Task：

| 目标角色 | Task 看板 | 初始状态 | 必填字段 |
|---------|----------|---------|---------|
| 张良 | 产品需求看板 | 待开始 | 需求来源、验收标准、优先级 |
| 韩信 | 研发任务看板 | 待开始 | 技术方案、预估工时、关联需求 |
| 曹参 | 迭代规划看板 | 待协调 | 范围变更说明、影响评估 |
| 郦食其 | 外部沟通看板 | 待执行 | 沟通要点、预期结果 |

**Task 创建后**：
1. 更新外部需求看板的「关联Task」字段
2. 更新意图分发日志的「执行状态」为「已通知」
3. 通知目标角色有新任务待处理
