## 一、负责维护的看板列表

| 看板名称 | 数据库类型 | 主要职责 | 访问权限 |
|---------|-----------|---------|---------|
| **迭代看板** | Story/Task 双层级 | 技术方案拆解、任务创建、代码审查跟踪 | 读写 |
| **技术债务看板** | 单层级任务 | 技术债务识别、优先级评估、偿还计划 | 读写更新 |

## 二、各看板操作权限与字段规范

### 2.1 迭代看板

#### 核心职责
- 将 Story 拆解为可执行的 Task
- 技术方案设计与文档化
- 代码审查与质量把控
- 14:00 研发联调状态同步

#### 技术方案 Task 字段规范

| 字段名 | 类型 | 必填 | 填写时机 | 填写规范 |
|-------|------|-----|---------|---------|
| `任务名称` | Title | 是 | 拆解时 | 【架构】技术方案名称 或 【实现】模块名称 |
| `类型` | Select | 是 | 创建时 | 技术方案/核心实现/代码审查/联调支持 |
| `状态` | Select | 是 | 全程 | 待协调/待开始/进行中/待验收/Bug/回流/已完成 |
| `负责人` | Person | 是 | 创建时 | 自己或韩信 |
| `Story 关联` | Relation | 是 | 创建时 | 关联父级 Story |
| `父级 Task` | Relation | 否 | 创建时 | 如有父级架构任务 |
| `技术方案链接` | URL | 是 | 方案完成后 | 指向 ADR 或技术文档 |
| `拆分任务清单` | Text | 否 | 拆解时 | 子任务列表，用 `- [ ]` 格式 |
| `代码审查链接` | URL | 否 | CR 时 | PR 链接 |
| `技术风险` | Text | 否 | 识别时 | 描述潜在技术风险 |
| `依赖项` | Text | 否 | 拆解时 | 列出前置依赖 |

### 2.2 技术债务看板

#### 职责定义
- 识别代码中的技术债务
- 评估债务影响与偿还优先级
- 制定偿还计划并跟踪
- 防止债务无序蔓延

#### 状态定义

| 状态值 | 说明 |
|-------|------|
| **已识别** | 债务已记录，待评估 |
| **评估中** | 正在评估影响范围和优先级 |
| **待排期** | 已评估，等待排入迭代 |
| **偿还中** | 正在处理中 |
| **已偿还** | 已完成偿还，经验证 |
| **已接受** | 经评估决定暂不处理 |

#### 字段填写规范

| 字段名 | 类型 | 必填 | 填写规范 |
|-------|------|-----|---------|
| `债务描述` | Title | 是 | 简洁描述技术债务 |
| `状态` | Select | 是 | 上述六种状态之一 |
| `债务类型` | Select | 是 | 代码债务/架构债务/测试债务/文档债务 |
| `严重程度` | Select | 是 | 致命/严重/一般/轻微 |
| `发现时间` | Date | 是 | 发现日期 |
| `发现人` | Person | 是 | 发现者 |
| `影响范围` | Text | 是 | 描述影响模块或功能 |
| `技术影响` | Text | 是 | 对性能/稳定性/可维护性的影响 |
| `预估偿还成本` | Number | 是 | 预估工时（小时）|
| `预估收益` | Text | 否 | 偿还后预期收益 |
| `偿还方案` | Text | 否 | 如何偿还此债务 |
| `计划偿还迭代` | Date | 否 | 计划在哪天迭代偿还 |
| `关联代码` | URL | 否 | 关联代码链接 |
| `备注` | Text | 否 | 其他补充信息 |

#### 技术债务评估矩阵

创建债务时，按以下矩阵评估严重程度：

| 影响/概率 | 高概率发生 | 低概率发生 |
|----------|-----------|-----------|
| **影响严重** | 致命 | 严重 |
| **影响一般** | 严重 | 一般 |
| **影响轻微** | 一般 | 轻微 |

#### 定期审查

- **每周五**：审查技术债务看板，决定下周偿还计划
- **每月末**：统计技术债务趋势，生成技术健康报告

### 2.3 Bug 看板（技术视角）

#### 核心职责
- 修复技术类 Bug
- 进行根因分析
- 防止同类问题再次发生

#### 技术 Bug 处理流程

```
新增 → 认领修复 → 根因分析 → 修复中 → 已修复 → 验证通过
```

#### 根因分析模板

修复 Bug 时，在备注中填写 RCA（Root Cause Analysis）：

```markdown
## 根因分析 (RCA)

### 问题现象
[描述 Bug 表现]

### 根本原因
[5 Whys 分析]

### 修复方案
[如何修复]

### 预防措施
[如何避免同类问题]

### 经验教训
[可复用的经验]
```

#### 技术 Bug 分类标签

| 标签 | 说明 |
|-----|------|
| `架构缺陷` | 架构设计导致的问题 |
| `接口问题` | API 定义或实现问题 |
| `性能问题` | 响应慢、资源消耗高 |
| `并发问题` | 多线程/并发相关问题 |
| `边界条件` | 未处理的边界情况 |
| `技术债务` | 历史遗留代码导致 |

### 2.4 架构决策记录(ADR)

#### 文档命名规范
`ADR-{序号:000}-{决策主题}.md`

示例：`ADR-001-数据库选型.md`

#### ADR 模板

```markdown
# ADR-XXX: 决策标题

## 状态
- [ ] 提议
- [ ] 已接受
- [ ] 已废弃
- [ ] 已取代（被 ADR-XXX 取代）

## 背景
[描述需要决策的问题背景]

## 决策
[明确决策内容]

## 备选方案

### 方案A：[名称]
- **优点**：
- **缺点**：

### 方案B：[名称]
- **优点**：
- **缺点**：

## 决策依据
[为什么选择当前方案]

## 影响
[此决策对系统的影响]

## 相关决策
- 依赖：ADR-XXX
- 被依赖：ADR-XXX

## 备注
[其他补充信息]

---
决策人：萧何
日期：YYYY-MM-DD
```

#### ADR 索引维护

在 `notion/architecture/ADR-INDEX.md` 维护所有 ADR 列表：

```markdown
# 架构决策记录索引

| 编号 | 标题 | 状态 | 日期 | 决策人 |
|-----|------|------|------|-------|
| ADR-001 | 数据库选型 | 已接受 | YYYY-MM-DD | 萧何 |
```

## 四、与 OpenViking 记忆的联动方式

### 4.1 记忆同步机制

| 联动方向 | 触发时机 | 操作内容 |
|---------|---------|---------|
| **技术方案 → OpenViking** | ADR 创建/更新 | 将架构决策同步到技术记忆 |
| **技术债务 → OpenViking** | 债务状态变更 | 更新技术债务清单记忆 |
| **OpenViking → Notion** | 记忆检索 | 参考历史决策辅助新方案设计 |

### 4.2 记忆沉淀位置

| 看板内容 | 沉淀位置 | 沉淀时机 |
|---------|---------|---------|
| 技术方案 | `memory/tech-designs/{task-id}.md` | 方案完成后 |
| 架构决策 | `memory/adr/ADR-XXX.md` | ADR 接受后 |
| 技术债务 | `memory/tech-debt/YYYY-MM.md` | 每月汇总 |
| 根因分析 | `memory/rca/BUG-{id}.md` | Bug 修复后 |
| 代码审查记录 | `memory/code-reviews/YYYY-MM-DD.md` | 每日汇总 |

### 4.3 关键技术记忆字段映射

```
技术方案 Task → OpenViking Memory
├── 任务名称 → memory.title
├── 技术方案链接 → memory.design_doc_url
├── 拆分任务清单 → memory.subtasks
├── 技术风险 → memory.risks
├── 依赖项 → memory.dependencies
└── 代码审查链接 → memory.code_review_url

技术债务 → OpenViking Memory
├── 债务描述 → memory.debt_description
├── 债务类型 → memory.debt_type
├── 严重程度 → memory.severity
├── 技术影响 → memory.technical_impact
├── 预估偿还成本 → memory.estimated_cost
└── 偿还方案 → memory.repayment_plan
```

### 4.4 记忆检索规范

- **方案设计前**：检索类似场景的历史 ADR
- **技术债务评估时**：检索同类债务的处理经验
- **Bug 修复时**：检索相关模块的历史 RCA
- **晨会时**：检索昨日技术任务的完成情况

## 七、协作接口

### 与韩信（研发执行）的协作
- 技术方案 Task 创建后，@韩信 分配实现任务
- 代码审查链接在 Task 中更新
- 联调阻塞及时在看板标记

### 与曹参（PMO）的协作
- 技术风险及时在看板标记为「待协调」
- 任务延期提前在进度对齐时（11:30）同步
- 技术债务偿还计划纳入迭代规划

### 与张良（产品经理）的协作
- 技术可行性评估反馈到 Story
- 需求设计问题 Bug 流转回产品侧
- 架构变更影响产品验收口径时及时同步
