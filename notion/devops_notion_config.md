## 一、职责定位

- **核心职能**：环境附着、部署到 OpenClaw、基座升级、安全加固、部署后验证
- **执行原则**：严格执行部署规范，确保系统稳定性
- **关键节点**：18:00 检查当日附着环境
- **升级机制**：阻塞超过 2 小时向曹参（PMO）升级

## 三、迭代看板（Task）操作规范

### 3.1 Task 相关操作（运维视角）

| 阶段 | 操作 | 状态列 | 说明 |
|------|------|--------|------|
| **部署准备** | 检查「待验收」任务 | 待验收 | 17:00 日终评审后确认部署清单 |
| **环境附着** | 更新 Task 环境信息 | 进行中 | 18:00 环境检查时记录 |
| **部署执行** | 创建部署单，关联 Task | 部署中 | 实际部署操作时 |
| **部署完成** | 更新 Task 状态 | 已完成 | 验证通过后 |
| **回滚处理** | 创建回滚记录 | Bug/回流 | 部署失败时 |

### 3.2 Task 必填字段（运维补充）

韩信负责填写基础字段，周勃需补充以下运维相关字段：

| 字段 | 填写时机 | 填写规范 |
|------|---------|---------|
| **部署环境** | 部署准备时 | staging / production |
| **部署版本** | 部署完成时 | 部署的代码版本号/Commit SHA |
| **部署时间** | 部署完成时 | 实际部署执行时间 |
| **回滚版本** | 创建回滚单时 | 回滚目标版本 |
| **环境 Token 开销** | 运维操作结束时 | 部署/回滚消耗的 token 数 |

## 四、部署发布看板操作规范

### 4.1 部署单创建规范

每个发布必须创建独立的部署单（Deployment Card）：

| 字段 | 必填 | 说明 |
|------|------|------|
| **部署单号** | 是 | 格式：DEP-{YYYYMMDD}-{序号} |
| **关联 Task** | 是 | 本次部署包含的所有 Task ID |
| **部署环境** | 是 | staging / production |
| **部署版本** | 是 | 代码版本/镜像标签 |
| **部署时间窗口** | 是 | 计划部署时间段 |
| **回滚方案** | 是 | 回滚步骤和回滚版本 |
| **验证检查项** | 是 | 部署后必须验证的功能点 |
| **风险等级** | 是 | 低/中/高/紧急 |

### 4.2 部署状态流转

```
┌─────────┐    ┌──────────┐    ┌─────────┐    ┌─────────┐
│ 待部署   │───→│ 部署中   │───→│ 部署完成 │───→│ 已验证  │
└─────────┘    └──────────┘    └─────────┘    └─────────┘
                                      │
                                      ▼
                               ┌──────────┐
                               │ 部署失败  │
                               │ →回滚中   │
                               └──────────┘
```

### 4.3 部署状态定义

| 状态 | 说明 | 负责人 |
|------|------|--------|
| **待部署** | 已通过验收，等待部署窗口 | 周勃 |
| **部署中** | 正在执行部署操作 | 周勃 |
| **部署完成** | 代码已发布，待验证 | 周勃/陈平 |
| **已验证** | 部署后验证通过，正式发布 | 陈平/张良 |
| **部署失败** | 部署过程出错 | 周勃 |
| **回滚中** | 正在执行回滚 | 周勃 |
| **已回滚** | 回滚完成，服务恢复 | 周勃 |

## 五、部署发布与看板状态更新流程

### 5.1 标准发布流程

```
1. 接收部署通知
   │
   ▼
2. 从「待验收」列确认待部署 Task
   │
   ▼
3. 创建部署单，填写部署信息
   │
   ▼
4. 更新部署单状态为「部署中」
   │
   ▼
5. 执行部署操作（OpenClaw 环境）
   │
   ▼
6. 部署完成，更新部署单为「部署完成」
   │
   ▼
7. 通知陈平进行部署后验证
   │
   ▼
8. 验证通过，更新为「已验证」，Task 移至「已完成」
```

### 5.2 紧急发布流程

```
1. 接收紧急发布通知（来自萧何/张良）
   │
   ▼
2. 创建紧急部署单，标记风险等级为「紧急」
   │
   ▼
3. 快速检查回滚方案
   │
   ▼
4. 执行部署，密切监控
   │
   ▼
5. 部署后立即验证核心功能
   │
   ▼
6. 验证通过，更新看板状态
```

### 5.3 回滚流程

```
1. 发现部署异常（监控告警/验证失败）
   │
   ▼
2. 立即评估是否需要回滚
   │
   ▼
3. 更新部署单状态为「回滚中」
   │
   ▼
4. 执行回滚操作
   │
   ▼
5. 验证服务恢复
   │
   ▼
6. 更新部署单为「已回滚」
   │
   ▼
7. 创建 Bug 卡记录问题（关联原部署单）
   │
   ▼
8. 通知相关角色（韩信/萧何）
```

### 5.4 看板状态同步要求

| 部署事件 | 看板操作 | 通知对象 |
|---------|---------|---------|
| 开始部署 | Task 添加标签 `#deploying` | 相关研发（韩信） |
| 部署完成 | 更新 Task 状态，填写部署信息 | 陈平（测试） |
| 部署失败 | Task 移至「Bug/回流」，创建 Bug 卡 | 韩信/萧何/曹参 |
| 开始回滚 | Task 添加标签 `#rollback` | 萧何/曹参 |
| 回滚完成 | 更新 Task 状态，记录回滚信息 | 全员 |

## 六、18:00 环境检查

### 6.1 检查内容

| 检查项 | 检查内容 | 看板记录 |
|--------|---------|---------|
| **环境附着状态** | OpenClaw 基座连接状态 | 环境状态看板更新 |
| **当前部署版本** | 各环境当前运行版本 | 版本号字段 |
| **待部署任务** | 已验收未部署的 Task | 确认部署计划 |
| **监控告警** | 系统监控状态 | 如有异常创建 Bug 卡 |
| **日志检查** | 关键错误日志 | 如有异常创建 Bug 卡 |

### 6.2 检查输出物

- 更新「环境状态看板」
- 向萧何汇报环境状态
- 如有异常，创建/更新 Bug 卡

### 6.3 环境状态记录模板

```markdown

### OpenClaw 基座状态
- 状态: 正常/异常
- 版本: vX.X.X
- 备注: xxx

### 各环境部署状态
| 环境 | 当前版本 | 最后部署时间 | 状态 |
|------|---------|-------------|------|
| staging | vX.X.X | YYYY-MM-DD HH:MM | 正常 |
| production | vX.X.X | YYYY-MM-DD HH:MM | 正常 |

### 待部署任务
- [ ] Task-001: xxx
- [ ] Task-002: xxx

### 异常事项
- [Bug-xxx] 问题描述
```

## 七、Bug 看板操作规范（运维视角）

### 7.1 运维相关 Bug 类型

| 类型 | 说明 | 处理流程 |
|------|------|---------|
| **部署失败** | 部署过程中出错 | 记录问题 → 回滚 → 创建 Bug |
| **环境问题** | 环境配置/基座问题 | 直接处理或升级 |
| **监控告警** | 系统监控触发告警 | 排查 → 修复/创建 Bug |
| **性能问题** | 线上性能异常 | 分析 → 创建 Bug 给研发 |

### 7.2 运维 Bug 状态流转

| 当前状态 | 运维操作 | 新状态 | 触发条件 |
|---------|---------|--------|---------|
| 新增 | 确认环境 Bug | **修复中** | 确认为运维负责的问题 |
| 新增 | 转给研发 | **新增** | 非环境问题，转给韩信 |
| 修复中 | 环境修复完成 | **已修复** | 环境问题处理完成 |
| 已修复 | 验证通过 | **已关闭** | 验证环境恢复正常 |

## 八、阻塞上报的看板记录方式

### 8.1 运维阻塞常见场景

- 部署权限不足
- 环境配置缺失
- 基座版本不兼容
- 依赖服务不可用
- 回滚版本缺失

### 8.2 阻塞上报流程

与研发流程一致，但重点关注部署相关阻塞：

```
发现部署阻塞
    │
    ▼
┌─────────────────────┐
│ 1. 在看板中标记       │
│    #deploy-blocked  │
└────────┬────────────┘
         │
    ┌────┴────┐
    ▼         ▼
  <2小时    >=2小时
    │         │
    ▼         ▼
尝试解决    向曹参升级
    │         │
    ▼         ▼
阻塞解除   记录在「待协调」列
继续部署   等待协调解决
```

### 8.3 阻塞标记规范

| 操作 | 说明 |
|------|------|
| **添加标签** | `#deploy-blocked` 或 `#env-blocked` |
| **在部署单中记录** | 阻塞原因、影响范围、预计影响时间 |
| **通知相关人员** | @ 萧何（技术负责人）和曹参（PMO） |
| **超过 2 小时** | 移至「待协调」列，晨会时重点提出 |

## 十、部署脚本与看板关联

### 10.1 部署脚本输出规范

部署脚本执行后，需输出以下信息便于看板记录：

```bash
# 部署完成后输出
DEPLOY_RESULT={
  "deploy_id": "DEP-20260327-001",
  "status": "success|failed|rolledback",
  "version": "v1.2.3",
  "commit_sha": "abc123",
  "start_time": "2026-03-27T18:00:00Z",
  "end_time": "2026-03-27T18:05:00Z",
  "environment": "staging|production",
  "related_tasks": ["TSK-001", "TSK-002"]
}
```

### 10.2 自动化看板更新（建议）

```bash
# 部署脚本中集成看板更新（如 API 可用）
after_deploy() {
  if [ $deploy_status == "success" ]; then
    # 更新 Task 状态为「已完成」
    update_notion_task "$task_id" "status" "已完成"
    update_notion_task "$task_id" "deploy_version" "$version"
    update_notion_task "$task_id" "deploy_time" "$end_time"
  else
    # 创建 Bug 卡
    create_notion_bug "$task_id" "部署失败: $error_msg"
  fi
}
```
