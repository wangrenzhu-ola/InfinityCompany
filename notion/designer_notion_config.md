## 一、设计工作看板结构

### 1.1 数据库名称
`🎨 设计任务看板` (Database)

### 1.2 视图配置

| 视图名称 | 类型 | 用途 |
|---------|------|------|
| 状态看板 | Board | 按状态分组跟踪设计任务 |
| 我的任务 | Table | 当前迭代的设计任务 |
| 待评审设计 | Table | 待产品/技术评审的设计稿 |
| 设计资产库 | Gallery | 已归档的设计规范和资源 |
| 按优先级 | Table | 按优先级排序 |
| 按迭代 | Board | 按迭代版本分组 |

## 二、设计任务字段填写规范

### 2.1 基础字段

| 字段名 | 类型 | 必填 | 填写规范 |
|--------|------|------|----------|
| **任务编号** | Title | ✅ | 自动生成或 `DES-YYYYMMDD-序号` |
| **任务名称** | Text | ✅ | 简明描述设计内容 |
| **状态** | Select | ✅ | 待排期 / 设计中 / 待评审 / 修改中 / 已确认 / 已归档 |
| **优先级** | Select | ✅ | 紧急 / 高 / 中 / 低 |
| **设计类型** | Select | ✅ | UI设计 / UX设计 / 交互设计 / 规范维护 / 图标/插画 |
| **指派给** | Person | ✅ | 叔孙通 |
| **关联需求** | Relation | ✅ | 关联产品需求单 |
| **关联迭代** | Select | ✅ | 所属迭代版本 |

### 2.2 设计内容字段

| 字段名 | 类型 | 必填 | 填写规范 |
|--------|------|------|----------|
| **设计目标** | Text | ✅ | 本次设计要解决的问题/达成的目标 |
| **设计范围** | Multi-select | ✅ | 页面/组件/流程图/原型/规范 |
| **设计稿链接** | URL | ✅ | Figma/Sketch/即时设计链接 |
| **设计说明** | Text | ✅ | 设计思路、交互逻辑说明 |
| **参考资源** | Files | 可选 | 竞品分析、灵感参考 |
| **技术限制** | Text | 可选 | 需要技术确认的实现约束 |

### 2.3 流程字段

| 字段名 | 类型 | 用途 |
|--------|------|------|
| **创建时间** | Date | 自动记录 |
| **预计完成** | Date | 计划交付日期 |
| **实际完成** | Date | 设计完成日期 |
| **评审时间** | Date | 设计评审会议日期 |
| **确认时间** | Date | 设计确认日期 |
| **评审人** | Person | 参与评审的产品/技术人员 |
| **修改次数** | Number | 设计修改轮次统计 |
| **最终设计稿** | Files | 确认后的设计导出文件 |

## 三、设计状态流转规则

### 3.1 状态定义

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  待排期  │ → │  设计中  │ → │  待评审  │ → │  已确认  │ → │  已归档  │
└─────────┘    └────┬────┘    └────┬────┘    └─────────┘    └─────────┘
                    │              │
                    └──────────────┘
                           ↓
                    ┌─────────┐
                    │  修改中  │
                    └─────────┘
```

### 3.2 状态流转说明

| 流转方向 | 触发条件 | 操作人 | 备注 |
|---------|---------|--------|------|
| 待排期 → 设计中 | 开始设计工作 | 叔孙通 | 每日站会后领取 |
| 设计中 → 待评审 | 设计初稿完成 | 叔孙通 | 在Notion评论@张良 |
| 待评审 → 已确认 | 评审通过 | 张良 | 无修改意见或确认修改后 |
| 待评审 → 修改中 | 评审需调整 | 张良 | 需注明修改意见 |
| 修改中 → 待评审 | 修改完成 | 叔孙通 | 再次提交评审 |
| 已确认 → 已归档 | 资产归档完成 | 陆贾 | 同步更新知识库 |

## 四、设计资产归档到知识库

### 4.1 归档范围

| 资产类型 | 归档内容 | 存储位置 |
|---------|---------|---------|
| **设计规范** | 色彩/字体/组件/间距规范 | Notion 知识库 - 设计规范 |
| **UI Kit** | 组件库、图标库源文件 | Notion 附件 + 云盘 |
| **页面设计** | 各功能页面设计稿 | Notion 知识库 - 页面设计 |
| **交互原型** | 流程图、原型链接 | Notion 嵌入 |
| **设计资源** | 切图、标注、标注文档 | Notion 附件 |
| **设计文档** | PRD配套设计说明 | Notion 页面 |

### 4.2 归档流程

```
设计确认(已确认状态)
        ↓
┌───────────────────┐
│  1. 整理设计资产   │
│  - 导出最终设计稿  │
│  - 整理切图资源    │
│  - 更新组件库      │
└─────────┬─────────┘
          ↓
┌───────────────────┐
│  2. Notion 录入    │
│  - 创建知识库页面  │
│  - 上传/嵌入设计稿 │
│  - 填写元数据      │
└─────────┬─────────┘
          ↓
┌───────────────────┐
│  3. 同步陆贾      │
│  - @陆贾 确认归档  │
│  - 更新设计资产库  │
└─────────┬─────────┘
          ↓
┌───────────────────┐
│  4. 状态更新      │
│  - 任务状态→已归档 │
│  - 更新完成时间    │
└───────────────────┘
```

### 4.3 知识库页面模板

**页面路径**: `知识库 > 设计资产 > [模块名] > [页面/功能名]`

页面内容结构：

```markdown
# [页面/功能名称] 设计稿

## 基本信息
- 关联需求: [需求链接]
- 设计版本: v1.0
- 设计日期: YYYY-MM-DD
- 设计师: 叔孙通

## 设计稿
[嵌入 Figma/设计工具]
或
[截图 + 附件]

## 设计说明
[交互逻辑、特殊状态说明]

## 切图资源
[切图附件或资源链接]

## 变更记录
| 日期 | 版本 | 变更内容 | 修改人 |
|------|------|---------|--------|
```

### 4.4 元数据规范

每个归档的设计资产页面必须包含：

| 元数据 | 填写位置 | 说明 |
|--------|---------|------|
| 模块分类 | Page 属性 | 所属产品模块 |
| 资产类型 | Page 属性 | UI/UX/组件/图标 |
| 关联迭代 | Page 属性 | 所属迭代版本 |
| 状态 | Page 属性 | 在用 / 废弃 / 更新中 |
| 标签 | Tags | 功能标签、版本标签 |

## 五、与产品经理的协作流程

### 5.1 需求接收

| 阶段 | 动作 | Notion操作 |
|------|------|-----------|
| 需求接收 | 张良创建设计任务 | 关联需求单，指派给叔孙通 |
| 需求澄清 | 同步会/评论沟通 | 在任务页面记录澄清结果 |
| 排期确认 | 站会确认迭代排期 | 更新"预计完成"日期 |

### 5.2 设计评审

| 步骤 | 动作 | 负责人 | Notion更新 |
|------|------|--------|-----------|
| 1 | 设计初稿完成 | 叔孙通 | 状态→待评审，@张良 |
| 2 | 预约评审时间 | 张良 | 更新"评审时间" |
| 3 | 设计评审会议 | 双方 | 评审意见记录在评论 |
| 4 | 评审结果确认 | 张良 | 状态→已确认/修改中 |

### 5.3 验收配合

- **验收角色**: 张良负责功能验收，叔孙通配合视觉验收
- **Bug关联**: 视觉类Bug需关联到对应设计任务
- **迭代看板**: 每日站会同步设计进度

## 六、每日定时任务 (14:00 看板更新)

### 6.1 定时任务触发

- **时间**: 每日 14:00
- **执行人**: 叔孙通
- **任务来源**: 系统定时提醒 / OpenViking 触发

### 6.2 操作清单

| 序号 | 操作项 | 具体操作 | 输出 |
|------|--------|---------|------|
| 1 | 检查"设计中"任务 | 确认当前设计进度，更新预估完成时间 | 进度备注 |
| 2 | 更新待评审任务 | 检查是否有待评审任务超时 | 催评提醒 |
| 3 | 整理修改任务 | 确认修改中的任务优先级 | 修改计划 |
| 4 | 更新设计资产 | 确认当日可归档的资产 | 归档清单 |
| 5 | 同步陆贾 | 通知知识库更新事项 | 协同确认 |

### 6.3 Notion 看板操作步骤

```
1. 打开 Notion 设计任务看板
2. 筛选"设计中"状态的任务
3. 更新各任务进度（在评论中添加进度%）
4. 检查"待评审"任务是否超时
   - 超时则在评论中 @张良 提醒
5. 检查"修改中"任务
   - 确认修改优先级
   - 更新预计完成时间
6. 查看"已确认"任务
   - 确认是否已完成资产归档
   - 未归档则加入今日待办
7. 在每日站会页面更新设计进度
```

## 七、设计规范维护

### 7.1 规范类型

| 规范类型 | 维护频率 | Notion存储位置 |
|---------|---------|---------------|
| 色彩规范 | 迭代更新 | 知识库 > 设计规范 > 色彩 |
| 字体规范 | 迭代更新 | 知识库 > 设计规范 > 字体 |
| 间距规范 | 迭代更新 | 知识库 > 设计规范 > 间距 |
| 组件规范 | 持续更新 | 知识库 > 设计规范 > 组件库 |
| 图标规范 | 持续更新 | 知识库 > 设计规范 > 图标库 |

### 7.2 规范更新流程

```
1. 识别规范变更需求（新功能/优化）
        ↓
2. 在 Notion 创建规范更新任务
   - 关联设计任务
   - 标记为"规范维护"类型
        ↓
3. 更新设计稿和规范文档
        ↓
4. 通知相关人员
   - @张良 产品确认
   - @韩信 技术确认可行性
        ↓
5. 确认后归档到知识库
   - 更新规范页面
   - 同步陆贾更新索引
        ↓
6. 通知团队规范变更
   - 在团队频道发布更新通知
   - 标注变更点和影响范围
```

## 八、常用 Notion 筛选器配置

### 8.1 保存的筛选器

```
📌 本周设计任务
   指派给 = 叔孙通
   AND 关联迭代 = 当前迭代
   AND 状态 != 已归档

📌 待评审设计
   状态 = 待评审
   AND 指派给 = 叔孙通

📌 高优先级未开始
   优先级 IN (紧急, 高)
   AND 状态 = 待排期

📌 待归档资产
   状态 = 已确认
   AND 实际完成 != 空

📌 设计规范维护任务
   设计类型 = 规范维护
   AND 状态 != 已归档
```

## 九、规范检查清单

每次提交设计评审前，确认：

- [ ] 设计目标明确
- [ ] 设计稿链接有效
- [ ] 交互说明完整
- [ ] 多状态/异常状态已考虑
- [ ] 响应式/适配方案已考虑
- [ ] 切图资源已准备（如需要）
- [ ] 设计符合现有规范
- [ ] 技术可行性已初步确认

每次归档设计资产前，确认：

- [ ] 设计已确认
- [ ] 设计稿已导出
- [ ] 知识库页面已创建
- [ ] 元数据填写完整
- [ ] 陆贾已同步
- [ ] 团队已通知（重要变更）
