# 韩信（全栈研发）Notion 看板使用规范

> 角色：全栈研发工程师（Full-Stack Developer）  
> 汇报对象：萧何（架构师/技术负责人）  
> 依据：DAILY_ROUTINE.md 规范  
> 最后更新：2026-03-27

---

## 一、职责定位

- **核心职能**：前后端 + AI 工作流三合一实现
- **执行原则**：按架构师（萧何）方案执行，不擅自变更技术方案
- **关键节点**：14:00 参与研发推进检查
- **升级机制**：阻塞超过 2 小时向曹参（PMO）升级

## 二、需要读写的看板

### 2.1 必读看板（只读）

| 看板名称 | 读取内容 | 读取时机 |
|---------|---------|---------|
| **Story 看板** | 产品需求、验收标准、Story 优先级 | 晨会后、开始新任务前 |
| **技术方案文档** | 萧何输出的技术设计、任务拆解 | 任务开始前 |
| **待协调看板** | 是否有阻塞自己的事项目前在协调中 | 晨会时 |

### 2.2 必写看板（读写）

| 看板名称 | 操作权限 | 主要职责 |
|---------|---------|---------|
| **迭代看板（Task）** | 创建/更新 Task，更新状态 | 任务执行与进度更新 |
| **Bug 看板** | 更新 Bug 状态（修复中/已修复） | Bug 修复状态流转 |

---

## 三、迭代看板（Task）操作规范

### 3.1 Task 生命周期操作

| 阶段 | 操作 | 状态列 | 说明 |
|------|------|--------|------|
| **领取任务** | 从「待开始」拖拽到「进行中」 | 进行中 | 14:00 前确认当日任务 |
| **开始执行** | 填写「开始时间」 | 进行中 | 实际开始工作时记录 |
| **进度更新** | 更新进度备注 | 进行中 | 11:30 进度对齐时更新 |
| **完成提交** | 代码提交 + 更新看板 | 待验收 | 自测通过后立即更新 |
| **Bug 修复** | 关联 Bug 卡，修复后更新 | 修复中→已修复 | 按 Bug 流程处理 |

### 3.2 必填字段规范

每个 Task 必须完整填写以下字段：

| 字段 | 填写时机 | 填写规范 |
|------|---------|---------|
| **Token 开销（预估）** | 任务开始时 | 根据任务复杂度预估，单位：千 tokens |
| **Token 开销（实际）** | 任务结束时 | 实际消耗的 token 数，用于成本核算 |
| **开始时间** | 任务开始时 | 实际开始工作的时间戳 |
| **计划结束时间** | 任务开始时 | 预估完成时间，需与萧何确认 |
| **实际结束时间** | 任务验收通过时 | 由张良验收后确认 |
| **关联 Story** | 任务创建时 | 必须关联对应的 Story ID |
| **代码分支** | 任务进行中 | 记录 feature/fix 分支名 |

### 3.3 WIP 限制

- 「进行中」列：**最多 2 个**任务
- 达到上限时，必须完成或移交现有任务才能开始新任务

## 四、Bug 看板操作规范

### 4.1 Bug 状态流转（韩信视角）

```
┌─────────┐    ┌──────────┐    ┌─────────┐    ┌─────────┐
│  新增   │───→│ 修复中   │───→│  已修复  │───→│  已关闭  │
│ (陈平)  │    │ (韩信)   │    │ (韩信)   │    │ (陈平)  │
└─────────┘    └──────────┘    └─────────┘    └─────────┘
      │                               │
      ▼                               ▼
┌─────────┐                   ┌──────────┐
│非Bug/需求│                   │ 重复打开  │
│设计问题  │                   │ (验证失败) │
└─────────┘                   └──────────┘
```

### 4.2 韩信可执行的状态变更

| 当前状态 | 韩信操作 | 新状态 | 触发条件 |
|---------|---------|--------|---------|
| 新增 | 开始修复 | **修复中** | 领取 Bug 开始处理 |
| 新增 | 判定为非 Bug | **非 Bug** | 与萧何确认后标记 |
| 新增 | 判定为设计问题 | **需求设计问题** | 流转给张良处理 |
| 修复中 | 修复完成提交 | **已修复** | 代码提交并自测通过 |
| 重复打开 | 重新修复 | **修复中** | 回归失败重新处理 |

### 4.3 Bug 卡必填字段

| 字段 | 填写规范 |
|------|---------|
| **关联 Task** | 必须关联引发 Bug 的原 Task |
| **修复分支** | 记录修复所用分支名 |
| **修复时间** | 记录开始修复和修复完成时间 |
| **修复 Token 开销** | Bug 修复消耗的 token 数 |
| **根因分类** | 代码缺陷/逻辑错误/边界情况/其他 |

## 五、阻塞上报的看板记录方式

### 5.1 阻塞判定标准

- 技术方案不明确
- 依赖外部接口/数据未就绪
- 环境问题导致无法继续
- 其他角色输出未按时交付

### 5.2 阻塞上报流程

```
发现阻塞
    │
    ▼
┌─────────────────┐
│ 1. 在看板中标记   │
│    阻塞原因      │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
  <2小时    >=2小时
    │         │
    ▼         ▼
等待解决   向曹参升级
    │         │
    ▼         ▼
阻塞解除   记录在「待协调」列
继续执行   等待协调解决
```

### 5.3 看板阻塞标记规范

| 操作 | 说明 |
|------|------|
| **在 Task 中添加标签** | `#blocked` |
| **在备注中记录** | 阻塞原因：xxx  阻塞时间：HH:MM  预计解决时间/依赖：xxx |
| **通知相关人员** | 在评论中 @ 相关角色（萧何/陈平/周勃等） |
| **超过 2 小时** | 将 Task 移至「待协调」列，并在晨会/进度对齐时提出 |

## 六、14:00 研发推进检查

### 6.1 检查内容

| 检查项 | 检查方式 |
|--------|---------|
| 当日任务进度 | 看板「进行中」Task 状态 |
| 是否有阻塞 | 检查 `#blocked` 标签和「待协调」列 |
| 联调依赖 | 与陈平确认测试准备情况 |
| 代码提交状态 | 确认 feature 分支最新提交 |

### 6.2 输出物

- 更新看板 Task 状态
- 向萧何汇报进度
- 暴露需要协调的阻塞点

## 七、每日工作节奏

| 时间 | 事项 | 看板操作 |
|------|------|---------|
| 09:30 | 晨会 | 确认当日任务，从「待开始」领任务 |
| 10:00 | 开始开发 | 更新 Task 为「进行中」，填写开始时间 |
| 11:30 | 进度对齐 | 更新任务进度备注 |
| 14:00 | **研发推进检查** | 向萧何汇报进度，暴露阻塞 |
| 17:00 | 日终评审 | 完成的任务移至「待验收」 |
| 17:20 | 看板归档 | 确认 Task 字段填写完整 |

## 八、代码提交与看板关联

### 8.1 Commit Message 规范

```
[Task-ID] 类型: 简述

详细描述（可选）

关联看板: https://notion.so/xxx
```

示例：
```
[TSK-20260327-001] feat: 实现用户登录 API

- 完成 JWT token 生成与验证
- 添加密码加密存储

关联看板: https://notion.so/task-xxx
```

### 8.2 分支命名规范

```
feature/{task-id}-{brief-desc}   # 新功能
fix/{task-id}-{brief-desc}       # Bug 修复
hotfix/{brief-desc}              # 线上紧急修复
```

## 九、质量检查清单

### 每个 Task 完成前自检

- [ ] 代码已提交到对应分支
- [ ] 本地自测通过
- [ ] Task 必填字段已填写完整
- [ ] Token 开销已记录实际值
- [ ] 已更新看板状态为「待验收」
- [ ] 如有 Bug，已关联 Bug 卡并更新状态

### 每日结束前检查

- [ ] 当日「进行中」任务已更新进度
- [ ] 阻塞事项已标记或上报
- [ ] 14:00 检查已向萧何汇报
- [ ] 看板状态与实际工作一致
