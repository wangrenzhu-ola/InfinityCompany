## 1. Notion API 属性类型映射表

| 属性类型 | API 名称 | 说明 | 示例值 |
|---------|---------|------|--------|
| 标题 | `title` | 页面主标题，每个数据库必须有一个 | "用户登录功能开发" |
| 富文本 | `rich_text` | 多行文本内容 | 描述、备注 |
| 单选 | `select` | 单选下拉框 | 优先级、状态 |
| 多选 | `multi_select` | 多选标签 | 标签、技能要求 |
| 状态 | `status` | 专门的状态字段（带分组功能） | 待处理、进行中、已完成 |
| 日期 | `date` | 日期时间选择器 | 2024-03-27 |
| 数字 | `number` | 数值类型，可配置格式 | Token 数量、工时 |
| 复选框 | `checkbox` | 布尔值 | true/false |
| URL | `url` | 链接地址 | https://... |
| 邮箱 | `email` | 邮箱地址 | user@example.com |
| 电话 | `phone` | 电话号码 | +86-xxx |
| 关联 | `relation` | 关联到其他数据库 | 数据库ID + 页面ID |
| 汇总 | `rollup` | 对关联数据的聚合计算 | count, sum, average |
| 创建时间 | `created_time` | 自动记录创建时间 | 2024-03-27T10:00:00Z |
| 最后编辑时间 | `last_edited_time` | 自动记录最后更新时间 | 2024-03-27T15:30:00Z |
| 创建者 | `created_by` | 自动记录创建用户 | User ID |
| 最后编辑者 | `last_edited_by` | 自动记录最后更新用户 | User ID |
| 公式 | `formula` | 基于其他字段的计算表达式 | prop("字段") * 2 |

## 2. 外部需求看板 (External Requirements)

**数据库用途**: 收集和管理来自外部渠道的需求，作为需求池的输入源

### 2.1 字段定义表

| 字段名 | 字段类型 | 是否必填 | 默认值 | 选项值 | 业务含义 |
|-------|---------|---------|--------|--------|---------|
| 需求标题 | `title` | ✅ | - | - | 外部需求的简要标题 |
| 需求描述 | `rich_text` | ✅ | - | - | 详细的需求描述内容 |
| 来源渠道 | `select` | ✅ | 未分类 | 客户反馈、市场调研、竞品分析、内部建议、其他 | 需求来源渠道 |
| 优先级 | `select` | ✅ | P2-普通 | P0-紧急、P1-重要、P2-普通、P3-低优 | 需求处理优先级 |
| 状态 | `status` | ✅ | 待评估 | 待评估、已评估、已采纳、已拒绝、已归档 | 需求处理状态 |
| 提交人 | `rich_text` | ❌ | - | - | 外部需求提交者信息 |
| 联系方式 | `email` | ❌ | - | - | 提交人联系方式 |
| 关联需求 | `relation` | ❌ | - | - | 关联到需求看板中的 Story |
| 创建时间 | `created_time` | ✅ | 自动 | - | 需求录入时间 |
| 最后更新时间 | `last_edited_time` | ✅ | 自动 | - | 最后更新时间 |

### 2.2 详细字段说明

#### 状态流转
```
待评估 → 已评估 → 已采纳 → [关联需求创建] → 已归档
  ↓        ↓        ↓
已拒绝 ←────────────┘
```

#### 优先级定义
- **P0-紧急**: 影响系统可用性的关键问题，需立即处理
- **P1-重要**: 重要功能需求，影响用户体验或业务目标
- **P2-普通**: 常规需求，按计划排期处理
- **P3-低优**: 优化类需求，资源充裕时处理

## 5. Bug 看板 (Bug Tracking)

**数据库用途**: 统一管理缺陷，跟踪缺陷从发现到修复的全生命周期

### 5.1 字段定义表

| 字段名 | 字段类型 | 是否必填 | 默认值 | 选项值 | 业务含义 |
|-------|---------|---------|--------|--------|---------|
| Bug 标题 | `title` | ✅ | - | - | Bug 简要描述 |
| 详细描述 | `rich_text` | ✅ | - | - | Bug 复现步骤、期望结果、实际结果 |
| 状态 | `status` | ✅ | 新增 | 新增、修复中、已修复、重复打开、非 Bug、需求设计问题 | Bug 处理状态 |
| 严重程度 | `select` | ✅ | 一般 | 致命、严重、一般、轻微、建议 | Bug 影响程度 |
| 优先级 | `select` | ✅ | P2 | P0-立即修复、P1-高优先级、P2-普通、P3-低优先级 | 修复优先级 |
| 发现人 | `rich_text` | ❌ | - | - | 发现 Bug 的人员/Agent |
| 负责人 | `rich_text` | ❌ | - | - | Bug 修复负责人 |
| 关联 Story | `relation` | ❌ | - | Story 数据库 | 关联的 Story |
| 关联 Task | `relation` | ❌ | - | Task 数据库 | 关联的 Task |
| 关联迭代 | `relation` | ❌ | - | 迭代看板 | 发现的迭代 |
| 复现环境 | `rich_text` | ❌ | - | - | 环境信息、版本号等 |
| 截图/日志 | `url` | ❌ | - | - | 相关截图或日志链接 |
| 发现时间 | `created_time` | ✅ | 自动 | - | Bug 创建时间 |
| 修复时间 | `date` | ❌ | - | - | Bug 修复完成时间 |
| 验证人 | `rich_text` | ❌ | - | - | 验证 Bug 修复的人员 |
| 修复说明 | `rich_text` | ❌ | - | - | 修复方案说明 |
| 最后更新时间 | `last_edited_time` | ✅ | 自动 | - | 最后更新时间 |

### 5.2 状态定义详解

| 状态 | 定义 | 流转条件 |
|-----|------|---------|
| **新增** | 新发现的 Bug，待分配和评估 | 创建后默认状态 |
| **修复中** | 已确认是 Bug，正在修复 | 从"新增"或"重复打开"转入 |
| **已修复** | Bug 已修复，待验证 | 修复完成后转入 |
| **重复打开** | 修复后验证未通过，重新打开 | 从"已修复"验证失败转入 |
| **非 Bug** | 经确认不是缺陷，如环境问题 | 从"新增"评估后转入 |
| **需求设计问题** | 实际是需求设计问题，非实现缺陷 | 从"新增"评估后转入 |

### 5.3 严重程度定义

| 级别 | 定义 | 响应要求 |
|-----|------|---------|
| **致命** | 系统崩溃、数据丢失、核心功能完全不可用 | 立即处理，阻塞发布 |
| **严重** | 主要功能异常，存在明显规避措施 | 24小时内修复 |
| **一般** | 次要功能异常，影响用户体验 | 当前迭代修复 |
| **轻微** | 界面细节问题，不影响功能 | 后续迭代优化 |
| **建议** | 优化建议，非缺陷 | 评估后决定是否采纳 |

## 6. 每日复盘 (Daily Retrospective)

**数据库用途**: 记录每日迭代回顾，沉淀产出、阻塞、风险和改进项

### 6.1 字段定义表

| 字段名 | 字段类型 | 是否必填 | 默认值 | 选项值 | 业务含义 |
|-------|---------|---------|--------|--------|---------|
| 复盘日期 | `title` | ✅ | - | - | 复盘日期，如 "2024-03-27 复盘" |
| 复盘日期(日期) | `date` | ✅ | - | - | 复盘日期字段 |
| 关联迭代 | `relation` | ✅ | - | 迭代看板 | 关联的迭代 |
| 当日产出 | `rich_text` | ✅ | - | - | 当日完成的主要产出 |
| 阻塞问题 | `rich_text` | ❌ | - | - | 遇到的阻塞和卡点 |
| 风险识别 | `rich_text` | ❌ | - | - | 识别的潜在风险 |
| 改进项 | `rich_text` | ❌ | - | - | 待改进的具体事项 |
| 明日计划 | `rich_text` | ❌ | - | - | 明日工作计划 |
| 团队感受 | `select` | ❌ | 正常 | 😀愉快、🙂正常、😐疲惫、😰压力大 | 当日团队状态 |
| 复盘状态 | `status` | ✅ | 进行中 | 待复盘、进行中、已完成、已归档 | 复盘记录状态 |
| 改进项数量 | `number` | ❌ | 0 | - | 记录的改进项数量 |
| 创建时间 | `created_time` | ✅ | 自动 | - | 复盘创建时间 |
| 最后更新时间 | `last_edited_time` | ✅ | 自动 | - | 最后更新时间 |

### 6.2 内容模板

#### 当日产出
```
- Story XXX: 完成 Task A、Task B
- Bug XXX: 已修复并验证
- 其他: ...
```

#### 阻塞问题
```
- [问题描述]: [当前状态/求助对象]
- 例: API 接口文档缺失: 等待后端提供
```

#### 风险识别
```
- [风险描述]: [影响范围] - [应对措施]
- 例: Token 开销超出预算: 影响后续任务 - 需优化提示词
```

#### 改进项
```
- [改进项]: [负责人] - [预计完成时间]
- 例: 建立代码审查清单: CodeReview-Agent - 本周五前
```

## 9. 数据库配置检查清单

创建数据库后，请检查以下配置：

### 通用配置
- [ ] 所有 `title` 字段已正确设置
- [ ] 所有 `status` 字段选项按顺序排列
- [ ] `created_time` 和 `last_edited_time` 自动记录已启用
- [ ] 视图筛选器已配置（如：按状态、按优先级）

### 关系配置
- [ ] 外部需求 → 需求看板: `关联需求` 关系字段
- [ ] Story → Task: `关联 Tasks` 双向关系
- [ ] Task → Story: `所属 Story` 双向关系
- [ ] Story/Task → 迭代: `所属迭代` 关系字段
- [ ] Bug → Story/Task/迭代: 关系字段
- [ ] 迭代 → 复盘: `关联复盘` 关系字段

### 汇总配置
- [ ] Story: `Task 数量` Rollup 配置正确
- [ ] Story: `已完成 Tasks` Rollup 配置正确（需筛选）
- [ ] Story: `完成进度` Formula 配置正确
- [ ] 迭代: `Task 总数` Rollup 配置正确
- [ ] 迭代: `已完成 Tasks` Rollup 配置正确
- [ ] 迭代: `进度百分比` Formula 配置正确
