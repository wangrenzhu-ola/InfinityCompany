## 🗂️ Notion 看板结构总览

根据 design.md 规范，公司使用以下 4 块核心看板：

```
┌─────────────────────────────────────────────────────────────┐
│                    Notion 企业知识库                         │
├──────────────────┬──────────────────┬───────────────────────┤
│   外部需求看板    │    迭代看板       │       Bug 看板        │
├──────────────────┴──────────────────┴───────────────────────┤
│                    每日复盘记录                              │
└─────────────────────────────────────────────────────────────┘
```

### 1. 外部需求看板
- **结构**: Story / Task 两层
- **关键字段**: 需求来源、登记时间、状态(待审/已批准/已拒绝/已归档)
- **流转链路**: 外部助理登记 → 私人助理读取(09:15) → Owner审核 → 分发执行

### 2. 迭代看板
- **结构**: 每天一个迭代，Story / Task 两层
- **关键字段**: Token开销、开始时间、计划结束时间、实际结束时间
- **使用角色**: 全团队，PMO主导维护

### 3. Bug 看板
- **结构**: 按缺陷单管理
- **状态集合**: 新增、重复打开、修复中、已修复、非Bug、需求设计问题
- **维护角色**: 测试(陈平) + 知识库管理员(陆贾)

### 4. 每日复盘记录
- **结构**: 每天一条记录
- **必填内容**: 当日产出、阻塞项、风险、改进项
- **组织者**: PMO(曹参) 19:00复盘会

## 🔗 与 OpenViking 记忆的联动

所有角色共享 OpenViking 基础记忆能力，与 Notion 看板的联动规则：

1. **Notion 为主**: 看板状态、任务进度以 Notion 为准
2. **OpenViking 为辅**: 历史上下文、决策依据、技术方案通过 OpenViking 检索
3. **同步机制**: 每日 11:00 陆贾校验同步状态
4. **记忆沉淀**: 复盘记录、技术方案、Bug根因分析同时入OpenViking
