# Agent 自动化基线（方案 B）

> 决策时间：2026-03-27
> 决策人：刘邦（Owner）批准，曹参执行
> 目的：梳理虚拟公司全员 Heartbeat 与 Cron，减少噪音，保留关键把控位。

## 一、治理原则

1. **Cron 管定点动作，Heartbeat 管周期巡检。**
2. **只保留 5 个关键角色开启 Heartbeat**：夏侯婴、郦食其、曹参、陈平、周勃。
3. **其余角色不开常规 Heartbeat**：张良、萧何、韩信、陆贾、叔孙通，仅通过 Cron 或任务触发拉起。
4. **Heartbeat 没事只回 `HEARTBEAT_OK`**，不重复汇报常规进度。
5. **阻塞超过 2 小时必须升级**；重大范围 / 优先级 / 排期变更必须请示 Owner。
6. **公司当前是 Owner 需求驱动型**，以低噪音、强控制、可追踪为第一原则。

## 四、消息与升级规则

### 默认静默

- 绝大多数 Cron 只做内部执行与留痕，不直接打扰 Owner。
- Heartbeat 仅在“有新风险 / 有新审批 / 有明显偏差”时发声。

### 直接提醒 Owner 的事项

- 夏侯婴：健康提醒、需要拍板的审批事项；
- 曹参：早会摘要、晚间复盘、阻塞超过 2 小时；
- 陈平：P0 / P1 缺陷、验收失败；
- 周勃：发布失败风险、回滚风险；
- 郦食其：高优先级外部事项、需审批外部请求。

## 五、配置备注

1. OpenClaw 中一旦任一 `agents.list[]` 定义 `heartbeat`，则**只有定义了 heartbeat 的角色会运行心跳**。
2. Heartbeat 统一采用轻上下文 + 独立会话，降低 Token 开销。
3. 韩信模型按 spec 修正为 `kimi-coding/k2p5`；MiniMax 继续作为全局兜底。
4. 该基线优先满足“低噪音、强节奏、可追踪”，后续如公司从需求驱动转为流水线驱动，再考虑扩大 Heartbeat 覆盖面。
