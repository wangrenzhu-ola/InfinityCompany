# InfinityCompany 仓库治理策略

## 适用范围

- InfinityCompany 是虚拟团队默认的日常研发主仓库
- 团队的代码改动、文档沉淀、流程优化、提示词演进、技能配置和交付物默认都必须在本仓库留痕
- 只有老板明确批准后，才允许把新项目拆分到独立 Git 仓库

## 权限分层

### 老板

- 拥有仓库最终决策权
- 拥有 `main` 分支保护策略调整权
- 拥有新项目分仓批准权

### 技术负责人

- 负责技术类变更的合并把关
- 负责脚本、配置、流程类改动的可用性验证
- 可以维护分支保护、标签和发布节奏

### 产品经理与 PMO

- 负责需求、迭代、验收、复盘相关文档和治理规则的完整性
- 对文档和流程变更拥有 review 责任

### 其他交付角色

- 默认没有直接推送 `main` 的权限
- 所有改动必须通过短生命周期分支和合并流程进入主线

## Trunk-Based 协作策略

- `main` 是唯一长期分支，始终保持可交付状态
- 所有改动必须从短生命周期分支发起
- 分支命名统一使用：
  - `feature/<topic>`
  - `fix/<topic>`
  - `docs/<topic>`
  - `ops/<topic>`
- 单个分支只解决一个明确问题，不同时承载多个主题
- 功能分支原则上应在当天或最迟下一工作日合并，避免长期漂移

## 合并门槛

- 禁止直接向 `main` 推送
- 合并前必须完成最小自检，至少包括：
  - 受影响脚本可执行或可解析
  - 受影响文档已同步更新
  - 受影响配置不包含本地敏感信息
- 涉及脚本、流程、治理规则的改动，必须至少经过一次 review
- 涉及知识结构、角色设定、提示词资产的改动，必须同步更新对应目录

## 默认留痕要求

### 代码留痕

- 脚本、配置、工作流、技能、提示词和自动化入口必须提交到仓库
- 本地临时修复如果影响长期运行，必须在当天回写到仓库

### 文档留痕

- 协作规则写入 `CONTRIBUTING.md`
- 仓库治理规则写入 `governance/`
- 知识结构、看板定义和模板写入 `notion/`
- 操作入口和启动方式写入 `README.md` 或相应操作文档

### 交付留痕

- 面向团队长期复用的交付物必须沉淀在仓库
- 部署、恢复、回滚入口必须以脚本形式进入仓库
- 外部项目如使用本仓库能力，必须能回溯到对应规范和脚手架来源

## 新项目分仓审批条件

只有满足以下条件之一，才允许新开独立 Git 仓库：

- 存在独立客户或外部合作方要求的代码隔离
- 存在安全隔离、权限隔离或合规要求
- 存在独立交付周期，且明显不适合在主仓库按目录维护

以下情况不能单独作为分仓理由：

- 个人习惯偏好
- 目录命名不满意
- 暂时不想整理主仓库结构
- 仅因为任务规模短期变大

## 分仓授权流程

- 申请人先填写 `governance/NEW_REPO_REQUEST.md`
- 在本仓库发起一条审批记录，说明分仓背景、边界、负责人、交付周期和回链计划
- 老板明确批准后，才允许创建新仓库并开放权限
- 未经批准擅自创建的新仓库，不视为正式交付仓库

## 回链规则

- 独立仓库必须在 `README.md` 明确声明其源自治理规则来自 InfinityCompany
- 通用脚手架、治理规则、共享提示词、公共脚本和流程模板必须优先回链到 InfinityCompany
- 任何在独立仓库沉淀出的通用能力，必须在合适时机同步回本仓库
- 独立仓库必须保留回链记录，至少包含：
  - 对应主仓库路径或文档
  - 负责人
  - 最近同步时间

## 验收检查点

- 是否存在未审批的独立仓库
- 是否存在只在本地运行、未回写主仓库的长期脚本
- 是否存在代码已变更但文档、看板、提示词未同步的情况
- 是否存在脱离 `main` 长期漂移的分支
