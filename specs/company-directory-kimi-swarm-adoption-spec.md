##1. 背景

当前公司已具备 `company-directory` 与 Kimi Swarm相关能力，但在实际协作中仍存在以下问题：

1. **协作口径不统一**：部分成员知道技能存在，但不知道何时用、怎么用、用到什么程度。
2. **跨部门沟通成本偏高**：跨 Agent 沟通仍依赖个人经验，容易出现找错人、消息未确认、升级路径不清晰的问题。
3. **复杂任务协同方式不稳定**：团队对 Kimi Swarm 的理解深浅不一，导致多角色并行任务时，输入输出格式不统一、执行方式不一致。
4. **验收标准缺失**：即便完成“学习”，也缺少可验证的掌握口径，后续很难判断是否真正能用于工作。

因此，需要发起一次**全员统一学习与实操验收**，把 `company-directory` 定义为**跨部门沟通基础能力**，把 Kimi Swarm 定义为**团队复杂任务协同助手能力**。

---

##2.需求目标

本需求希望达成三个结果：

1. **统一沟通底座**：全员知道应该找谁、如何联系、联系后如何做双向确认。
2. **统一协同方法**：全员理解 Kimi Swarm 的适用场景、输入方式、产出要求与降级路径。
3. **形成可验收的组织能力**：不是“听过”，而是“能说清、能演示、能留痕”。

##4.需求说明

###4.1 `company-directory` 学习要求

全员需要掌握以下内容：

1. **知道它是什么**
 - 是公司成员目录与标准化通讯协议
 - 用于查询人员、岗位、汇报线、升级路径、联系方式

2. **知道它解决什么问题**
 - 不知道该找谁
 - 不知道该用同步还是异步沟通
 - 消息发出后缺少确认闭环
 - 遇到异常时不知道怎么升级

3. **知道标准使用方式**
 - 实时沟通：`acpx-infinity`
 - 异步沟通：`emergency_inbox`
 - 接收消息后需做**双向确认**：复述上下文 + 明确接收状态

4. **知道如何选择沟通方式**
 -需要即时确认、多轮互动、紧急沟通 → `acpx-infinity`
 -需要派发长期任务、大段说明、对方暂时不在线 → `emergency_inbox`

5. **知道异常时怎么处理**
 - 联系失败时，切换到异步邮件
 - 超过约定时间未响应时，按升级路径处理
 - 不得把未确认的消息视为已送达

###4.2 Kimi Swarm 学习要求

全员需要掌握以下内容：

1. **知道它是什么**
 - Kimi Swarm 是团队处理复杂任务的协同助手能力
 -适用于多角色并行、弱依赖拆分、需要汇总产出的任务

2. **知道什么场景该用**
 -需要3 个及以上角色共同推进
 - 子任务之间可以并行
 -需要统一汇总多个角色产出
 -需要提高复杂任务推进效率

3. **知道什么场景不该强行用**
 -任务规模很小、单人即可完成
 - 强顺序依赖明显，前一步不完成后一步无法开始
 -需求本身尚未澄清，过早并行会放大返工成本

4. **知道输入应该怎么写**
 一个合格的 Swarm 输入，至少应包含：
 - 主任务目标
 - 子角色分工
 - 每个子任务的输出物
 - 时间要求/优先级
 - 汇总方式

5. **知道输出应该长什么样**
 - 每个角色有独立产出
 - 主控方有汇总结果
 - 明确已完成、待确认、风险项、依赖项

6. **知道降级路径**
 - 不适合 Swarm 时，应回退为普通单 Agent 执行 + PMO 协调
 - 如果底层通信能力异常，应说明降级方案，而不是假定任务已成功分发

###4.3口径统一说明（重要）

为避免团队继续在概念上“各说各话”，本需求明确以下统一口径：

####4.3.1 `company-directory` 的验收口径
- **以 `skills/company-directory/SKILL.md` 为准**
- 实时通讯统一按 **`acpx-infinity`**认定
- README 中旧示例若仍出现 `acpx`，视为历史兼容说明，**不作为正式验收口径**

####4.3.2 Kimi Swarm 的验收口径
- 验收关注的是**Swarm 能力是否被正确理解和使用**，而不是死记某一个路径名
- 可接受的学习对象包括：
 - 团队当前采用的 Swarm 能力说明
 - `skills/kimi-swarm-acpx/README.md`
 - 经审批采用的 `kimi-agent-swarm`相关参考材料
-只要成员能说清：**何时用、怎么拆、如何汇总、失败如何降级**，即可视为理解到位

###4.4 留痕要求

所有学习与验收必须留痕，至少满足以下之一：

1. 提交到各自 Agent目录下的学习记录/实践记录；或
2.记录到 Notion 看板；或
3. 在统一验收文档中登记完成状态与证据链接。

不接受“口头说会了但没有任何记录”的完成方式。

##5. 非目标

本需求**不要求**：

1. 全员都去实现或改造 `company-directory` / Kimi Swarm 的底层代码
2. 全员都具备架构级或开发级的 Swarm 调度能力
3. 用技能学习替代原有角色边界、审批流程和 PMO 管理流程

换句话说，这次要建立的是**组织协作能力**，不是把每个人都训练成底层框架维护者。

#### E.口径一致性
- [ ] 后续对内培训/答疑中，不再把 `sessions_send`作为 `company-directory` 的官方标准答案

###6.2 可接受证据

以下证据可作为验收材料：

- 学习笔记
- 演练记录
- 命令示例与执行回执
- 对话截图/会话记录
- Swarm任务模板

###6.3 判定失败的典型情形

出现以下任一情形，则不能判定为通过：

1.只知道名字，不知道适用场景
2.只会背命令，不知道何时该升级、何时该降级
3.发送了消息但没有确认回复，仍声称“已完成沟通”
4.说自己会用 Swarm，但无法写出最基本的任务拆分模板
5. 完成学习但没有留痕

##8.依赖与风险

###8.1依赖
- `company-directory` 文档和入口可访问
- Kimi Swarm参考材料可访问
- 跨 Agent 演练环境可用

###8.2 风险
- 不同文档中的命令示例不一致，容易让成员误判
- 部分角色对 Swarm 的理解偏“技术化”，导致非技术岗位畏难
- 如果只做宣导、不做演练，最终会停留在“知道有这个东西”阶段

###8.3 风险应对
-统一以本需求与正式 Skill 文档为口径
- 对非技术角色强调“会发起、会判断、会验收”，而非“会实现底层”
- 学习与演练绑定，不单独验收“理论掌握”

---

##9.结论

这项需求的本质，不是让团队多学两个新名词，而是把**沟通协议**和**复杂任务协同方式**真正变成组织共识。

如果说 `company-directory` 是公司的“道路与门牌”，那么 Kimi Swarm 就像“多兵种协同作战的阵法”。前者解决“找到谁、怎么说”，后者解决“多人如何同时动起来且不打架”。

只有两者一起建立起来，后续跨部门协作才会真正顺畅。
